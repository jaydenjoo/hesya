AI에게 검증을 요청할 때 — 실수 차단 protocol

AI는 "100% 일치"라고 자신 있게 보고하면서도 실제로는 검증을 부실하게 했을 수 있습니다. Jayden이 다음 5가지 protocol을 채팅창에 적용하면 같은 사고를 차단합니다.

📖 어떻게 사용하나요?

AI에게 검증·구현·디자인 정합 작업을 요청할 때, 아래 5가지 protocol 중 해당하는 것의 ✅ 강한 지시 박스를 그대로 채팅창에 복사 + 붙여넣기 하세요. AI가 검증 결과를 토큰 단위로 보고하도록 강제됩니다.

5가지 Protocol
01

"100% 정합" 검증 요청 시 — 근거 데이터 첨부 의무화

왜 필요한가: AI는 "100% 일치"를 자신 있게 보고하면서 실제로는 일부 요소만 검증했을 수 있음. 검증 결과를 표 형태로 강제하면 모호한 보고 차단.

💡 비유 — 화가에게 "그림 확인해줘" 요청 vs "색·구도·디테일 항목별 평가해줘"의 차이. 후자는 "예쁘다"는 모호한 답변이 불가능합니다.
❌ 약한 지시

"디자인이랑 100% 같은지 확인해줘"

✅ 강한 지시

토큰 단위 표로 검증해줘 — 폰트·색·spacing 항목별. 근거 없는 "일치" 보고 금지.

복사용 prompt template 
이 페이지가 디자인 파일과 100% 일치하는지 검증해줘.
다음 토큰을 모두 표로 만들어 비교 결과 보여줘:

1. 폰트 — element별 family / weight / style / size / line-height
2. 색 — 모든 hex 값 (배경·텍스트·border·아이콘)
3. spacing — margin / padding 픽셀 단위
4. radius — border-radius 모든 값
5. shadow — 박스 그림자 정의
6. motion — transition·animation duration / easing

각 항목마다 "디자인 파일 값 → 현재 코드 값 → 일치 ✅ / 불일치 ❌"
형식으로 표 작성. 근거 데이터 없이 "일치" 보고 금지.
02

AI 보고 받을 때 — "실제로 어디까지 검증했는가?" 질문

왜 필요한가: AI가 "Reference 100% TRUE 정합 (96%+)" 같은 모호한 표현을 사용할 때, 실제 검증 범위와 미검증 범위를 분리해 표시하도록 강제. 정직성 확보.

💡 비유 — 시험 채점 후 "잘 봤어"가 아니라 "어떤 문제 맞혔고 어떤 문제 안 풀었는지" 채점표 보여달라는 것.
❌ 받지 말아야 할 보고

"Reference 100% 정합 완료"

✅ 받아야 할 보고

"검증함 N개 / 미검증 M개 / 차이 발견 K개" 명시 + 미검증 사유

AI 보고 받고 확인용 질문 
방금 보고에서:

1. 실제로 검증한 항목은 무엇이고 미검증한 항목은 무엇인가?
   각각 개수 + 항목명 명시.
2. "100% 정합"이라고 한 근거는 무엇인가?
   각 element별 토큰 비교 결과를 표로 보여줘.
3. 어떤 부분은 visual diff 도구 없이 "추정"이고 어떤 부분은
   "코드 직접 비교"인가? 구분해서 표기.
4. 검증 누락된 영역 (예: 외부 import 파일·반응형 breakpoint·
   호버 상태·dark mode)이 있으면 명시.
03

Reference 파일 외부 의존 — 함께 보도록 요청

왜 필요한가: Reference HTML 파일에는 <link href="tokens.css">, @import, <script src> 같은 외부 의존이 있음. 인라인 CSS만 보면 폰트 weight·색 정의 등을 놓침. 이번 사고의 직접 원인.

💡 비유 — 영수증 위쪽만 보지 말고 첨부된 영수증·세금 명세서도 함께 봐달라는 것. 한 페이지만 보면 중요한 숫자를 놓침.
❌ 약한 지시

"이 HTML 파일이랑 비교해줘"

✅ 강한 지시

HTML 파일 + 모든 외부 의존 (tokens.css·.css·.jsx·image) 함께 비교

복사용 prompt template 
이 reference 파일과 현재 구현을 비교 검증할 때 반드시 함께 읽어줘:

1. Reference HTML의 다음 외부 의존 파일 모두:
   - <link rel="stylesheet" href="..."> — 외부 CSS
   - @import "..." — CSS 안의 import
   - <script src="..."> — JS 파일
   - <img src="..."> — 이미지 (디자인 의도 확인)
2. 같은 폴더의 .css / .jsx 파일도 함께 — 디자이너가 명세를
   여러 파일로 분리했을 수 있음.
3. 인라인 CSS만 보고 "검증 완료" 보고 금지.
4. 외부 파일 못 읽으면 "이 파일을 못 읽어 미검증" 명시.
04

같은 영역 PR 3개+ 누적 시 — 회고 강제

왜 필요한가: 같은 페이지·기능에 fix PR이 3개+ 누적되면 plan 자체가 부실했다는 신호 (L-082 규칙). AI가 자체 trigger 안 할 수 있으니 Jayden이 명시 요청.

💡 비유 — 같은 벽을 세 번 페인트 칠하면 칠 방법 자체가 잘못된 것. 페인트 종류·도구 다시 점검 필요.
회고 요청 prompt 
같은 영역(예: marketing landing / store dashboard)에 PR이
3개 이상 누적됐어. 회고해줘:

1. 왜 처음 plan에서 이 변경들을 한 번에 못 잡았나?
   (정보 부족 / 검증 부실 / 가정 오류 / 다른 이유)
2. 다음 plan부터는 어떻게 prevention 할 수 있는가?
3. 이 영역 작업 중단하고 plan 자체를 다시 만들어야 하는가?
4. PROGRESS.md / docs/learnings.md에 기록할 교훈 정리.
05

법규·정직성·신뢰 영역 — 별도 검증 protocol

왜 필요한가: Hesya는 🔴 RED 프로젝트 (돈·신분·법). AI 생성 콘텐츠 라벨링, 가짜 후기 금지 등 법규는 enforcement 진행 중. "괜찮을 거다" 추측 금지, 최신 정보 검증 의무.

💡 비유 — 의사가 환자에게 약 처방 시 "아마 괜찮을 것" 추측 X. 약 상호작용·알레르기·임신 여부 모두 확인 후 처방.
법규·신뢰 검증 prompt 
이 변경이 법규·신뢰성 영역과 관련될 수 있는지 검증해줘:

1. AI 생성 콘텐츠 (이미지·비디오·후기) 사용 → 라벨링 의무
   - 🇰🇷 한국 AI 광고법 (2026-01-22 발효)
   - 🇪🇺 EU AI Act Article 50 (2026-08-02 발효)
   - 🇺🇸 FTC Fake Reviews Rule (2024-10-21 발효)
2. 가짜 후기·가짜 SNS 출처 표기 → FTC 위반
3. 결제·인증·KYC 영역 → 보안 등급 🔴 확인
4. WebSearch 도구로 오늘 날짜 기준 최신 enforcement 사례
   1건 이상 확인 후 보고. 추측 금지.
Bonus — 정직성 강제 규칙
B-01

"확신등급" 표시 의무화

AI 응답 끝에 정보 출처 명확히 표기 — 추측·추정·검증의 경계를 시각적으로 분리.

매 응답에 표시 의무 
이 응답을 어떻게 검증했는지 끝에 표시해줘:

🟢 공식 문서·코드 직접 확인 = 확실
🟡 추정·기존 지식·문서 인용만 = 검증 필요
🔴 화면 캡처 또는 실제 측정 필요 = 추측

각 부분에 어떤 등급이 적용되는지 명시.
B-02

측정·시연 결과만 인정 (L-082)

"코드 머지 완료" ≠ "기능 작동". % 진척은 e2e 시연 가능 여부 기준만 인정.

진척 보고 시 강제 
이 기능이 e2e 시연 가능한지 검증해줘:

1. 외부인이 본인 컴퓨터에서 같은 결과 시연 가능한가?
2. 본인 환경(.env.local·로컬 DB)에만 의존하면 시연 불가.
3. Vercel preview / production에서 시연 가능 시점이 언제인가?
4. 미시연 = 진척률 0%. 코드 머지만 했으면 "스키마만 완료"로 표기.
📚 참고
  • 이 protocol은 ~/.claude/CLAUDE.md (글로벌) 의 검증 정책과 Hesya/CLAUDE.md (프로젝트) 의 Pre-Plan Inventory 절차에서 도출됨.
  • L-082 자기평가 규칙: % 표시 = e2e 시연 가능 여부 기준. 코드 머지 완료 ≠ 시연 가능.
  • 같은 영역 PR 3개+ 누적 시 즉시 회고 trigger 의무 (Plan 인벤토리 부실 시그널).
  • P0 Epic은 senior-engineer + code-explorer subagent로 객관화.
